# Right-Sizing

The default failure mode to avoid is overdesign.

## Default posture

- Solve the task at the smallest complexity level that still makes execution reliable.
- Prefer direct scope, non-goals, and deliverables over architecture and process.
- Keep optional hardening, automation, or extensibility out of the main plan unless the user asked for them.

## Common overdesign traps

- Treating a local or one-off task as if it needs full project scaffolding
- Adding heavy governance, defensive process, or long future-roadmap sections
- Inflating a clarification task into a redesign task
- Converting small uncertainties into broad risk-management overhead

## Useful self-checks

- Does this plan add a layer the user did not ask for?
- Which parts could be removed without harming execution?
- Are the stated risks practical, or just theoretically possible?
